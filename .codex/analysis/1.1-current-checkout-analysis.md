# 1.1 Current Checkout Analysis

Date: 2026-06-10

Scope: OpenCart 3.0.3.2 storefront checkout, cart AJAX, checkout-related payment/shipping/total extensions, checkout events, API-adjacent checkout controllers. Generated modification directories were not audited as primary source.

## Summary

Checkout is the standard OpenCart 3 accordion checkout implemented by `catalog/controller/checkout/*` and inline JavaScript in `catalog/view/theme/default/template/checkout/checkout.twig`.

The browser flow is AJAX-driven:

1. `checkout/checkout` renders empty accordion panels.
2. The page loads either `checkout/login` for guests or `checkout/payment_address` for logged customers.
3. Customer identity/address steps populate session state.
4. Shipping methods are calculated from `extension/shipping/*` models.
5. Payment methods are calculated from `extension/payment/*` models.
6. `checkout/confirm` builds `order_data`, creates/updates `session->data['order_id']`, and loads the selected payment controller.
7. The selected payment controller confirms or redirects to the gateway, then success/failure pages finalize the user flow.

Important custom checkout logic exists in CDEK, Measoft, `cod_cdek`, `filterit`, `sms_alert`, Yandex buy status events, and voucher email events.

## All Checkout AJAX Routes

- `POST index.php?route=checkout/shipping_address/save` - save selected or new shipping address.

## Current Checkout Step Sequence

### Entry Conditions

1. `checkout/checkout::index` redirects to `checkout/cart` if cart has no products/vouchers or stock rules fail.
2. It validates minimum quantity for each cart product.
3. It loads language, breadcrumbs, scripts, account mode, `shipping_required`, and layout controllers.
4. It renders `checkout/checkout.twig`.

### Guest Not Logged In

1. Page ready loads `checkout/login` into `collapse-checkout-option`.
2. User selects `register` or `guest`; `button-account` loads `checkout/register` or `checkout/guest` into `collapse-payment-address`.
3. If user logs in, `checkout/login/save` authenticates and redirects back to `checkout/checkout`.
4. Register path posts `checkout/register/save`.
5. Guest path posts `checkout/guest/save`.
6. If shipping is required and payment address is also shipping address, checkout loads `checkout/shipping_method` and additionally refreshes `checkout/shipping_address` or `checkout/guest_shipping`.
7. If shipping is required and a separate shipping address is needed, checkout loads `checkout/shipping_address` or `checkout/guest_shipping`.
8. If shipping is not required, checkout loads `checkout/payment_method`.

### Logged Customer

1. Page ready loads `checkout/payment_address`.
2. `button-payment-address` posts `checkout/payment_address/save`.
3. If shipping is required, checkout loads `checkout/shipping_address`.
4. `button-shipping-address` posts `checkout/shipping_address/save`, then loads `checkout/shipping_method`.
5. If shipping is not required, checkout loads `checkout/payment_method`.

### Shipping Method

1. `checkout/shipping_method::index` loads enabled shipping extensions from `model_setting_extension->getExtensions('shipping')`.
2. For each active shipping extension it calls `model_extension_shipping_{code}->getQuote($session->data['shipping_address'])`.
3. It stores sorted results in `session->data['shipping_methods']`.
4. `button-shipping-method` posts selected method/comment to `checkout/shipping_method/save`.
5. On success it loads `checkout/payment_method`.
6. CDEK may intercept `checkout/shipping_method/save/before` and require selected PVZ.

### Payment Method

1. `checkout/payment_method::index` recalculates totals through enabled total extensions.
2. It loads enabled payment extensions from `model_setting_extension->getExtensions('payment')`.
3. For each active payment extension it calls `model_extension_payment_{code}->getMethod($session->data['payment_address'], $total)`.
4. It stores sorted results in `session->data['payment_methods']`.
5. `button-payment-method` posts selected method/comment/terms agreement to `checkout/payment_method/save`.
6. On success it loads `checkout/confirm`.

### Confirm And Order Creation

1. `checkout/confirm::index` verifies shipping and payment session prerequisites.
2. It recalculates totals through enabled total extensions.
3. It builds `order_data` from customer/guest data, payment address, shipping address, cart products, vouchers, comments, tracking, currency, IP and user agent.
4. It calls `model_checkout_order->addOrder($order_data)` and writes `session->data['order_id']`.
5. It builds display products/totals for `checkout/confirm.twig`.
6. It loads selected payment controller using `extension/payment/{payment_method.code}`.
7. The payment template/controller confirms order history immediately or redirects/submits to external gateway.

### Success/Failure

1. Successful payment routes to `checkout/success`.
2. `checkout/success::index` clears cart, order/session checkout data, coupon/voucher/reward state.
3. CDEK success event also clears `session->data['cdek']`.
4. Failed external payment routes can use `checkout/failure`.

## Extension Points For New Logic

### Low-Risk Extension Points

- Add an OpenCart event before/after `catalog/controller/checkout/payment_method/save` to validate payment selection.
- Add an OpenCart event before/after `catalog/controller/checkout/shipping_method/save` to validate shipping selection.
- Add an OpenCart event before/after `catalog/model/checkout/order/addOrder` to enrich order data after creation.
- Add an OpenCart event before/after `catalog/model/checkout/order/addOrderHistory` to trigger integrations after payment/order status changes.
- Add a new `catalog/model/extension/total/{code}.php` to calculate fees/discounts using the existing totals pipeline.
- Add a new `catalog/model/extension/payment/{code}.php` and `catalog/controller/extension/payment/{code}.php` for payment-specific logic.
- Add a new `catalog/model/extension/shipping/{code}.php` for shipping rates, using `getQuote($address)`.

### Frontend Extension Points

- `catalog/view/theme/default/template/checkout/checkout.twig` is the central client state machine. Any new AJAX step must account for all branches: logged customer, guest, register, shipping required, shipping not required.
- `checkout/payment_method.twig` is the safest place to add UI tied to payment selection.
- `checkout/shipping_method.twig` is the safest place to add UI tied to shipping selection.
- `checkout/confirm.twig` is the safest place to add final read-only order summary UI.
- Address templates can add new fields, but corresponding `save` controllers must validate and persist them into session/order data.

### Session State Extension Points

- `session->data['payment_address']`
- `session->data['shipping_address']`
- `session->data['payment_methods']`
- `session->data['payment_method']`
- `session->data['shipping_methods']`
- `session->data['shipping_method']`
- `session->data['guest']`
- `session->data['comment']`
- `session->data['order_id']`
- `session->data['cdek']`

### Existing Custom Logic To Preserve

- CDEK stores PVZ data in `session->data['cdek']`, validates it before shipping method save, and writes it to `order_to_sdek` plus order history comment.
- `cod_cdek` payment availability depends on CDEK/shipping settings and may force `session->data['payment_method']['code'] = 'cod_cdek'`.
- `total/cdek` adds or subtracts a percent-based fee when `payment_method.code == cod_cdek`.
- Measoft JS and controller can mutate `session->data['shipping_method']` directly for `measoftcouriershipping.standard`.
- `filterit` has payment/total logic that can alter costs based on selected payment method.
- `sms_alert`, Yandex buy modules, and voucher total register events on `addOrderHistory/after`.

### Risk Notes

- `catalog/controller/event/cdekshipping.php` uses assignment in route checks: `if($route = "checkout/order/addOrder" && ...)`. This should be treated carefully if modifying event logic because it does not compare the route string.
- `catalog/model/extension/payment/modulbank.php` appears to query `payment_liqpay_geo_zone_id` while checking Modulbank availability. This may be a copy-paste bug affecting payment visibility.
- `catalog/model/extension/total/cdek.php` writes full session data to `total_cdek` log during total calculation. This can leak operational/customer data in logs.
- `system/storage/modification/` and `storage79/modification/` may contain generated runtime variants. They were intentionally not used as the source of truth.
