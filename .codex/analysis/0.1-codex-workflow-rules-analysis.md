# 0.1 Анализ Правил Рабочего Процесса Codex

## Текущее Поведение

`.codex/PROJECT_STRUCTURE.md` уже задает карту репозитория, область поиска и директории, которых нужно избегать. Существующие документы `.codex` уже задают четырехпапочную структуру документации задач, но пока не делают каждое требование пользователя явным.

## Релевантные Файлы

- `.codex/README.md`
- `.codex/RULES.md`
- `.codex/CHECKLISTS.md`
- `.codex/analysis/README.md`
- `.codex/implementation/README.md`
- `.codex/reports/README.md`
- `.codex/specs/0.1-codex-workflow-rules.md`
- `.codex/analysis/0.1-codex-workflow-rules-analysis.md`
- `.codex/implementation/0.1-implementation-log.md`
- `.codex/reports/0.1-final-report.md`

## Результаты Анализа

- Запрошенное поведение является долговечным знанием о workflow, поэтому ему место в `.codex/RULES.md` и `.codex/CHECKLISTS.md`.
- README-файлы отдельных папок должны документировать локальные ожидания для `analysis/`, `implementation/` и `reports/`.
- Текущая задача должна использовать тот же workflow документации, который формализуется.
- Новое требование о русском языке относится ко всей будущей документации `.codex`, поэтому его нужно закрепить в общих правилах и README подпапок.

## Решения

- Использовать task id `0.1`, потому что это задача workflow/meta-documentation репозитория, а не feature-задача OpenCart.
- Сохранять существующую документацию и добавлять новые правила вместо полного переписывания `.codex` docs.
- Считать правки документации изменениями, которые нужно перечислять в implementation log.
- Технические идентификаторы, пути и код не переводить, чтобы документы оставались применимыми к репозиторию.

## Архитектурные Решения

- Оставить `.codex/RULES.md` authoritative source для обязательных правил workflow.
- Оставить `.codex/CHECKLISTS.md` executable checklist, используемый до, во время и после работы.
- Хранить task-specific details в `specs/`, `analysis/`, `implementation/` и `reports/`, чтобы не перегружать глобальную карту проекта.

## Риски

- Больше обязательной документации добавляет overhead для небольших задач.
- Workflow нужно применять к нетривиальным задачам; короткие подтверждения или простые ответы могут не требовать полного набора из четырех task files.
- При переводе важно не менять технические пути, маршруты и имена классов.
