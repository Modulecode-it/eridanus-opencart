# Ядро Системы Памяти Агента

Последнее обновление: см. `CHANGELOG.md`

Этот файл описывает **универсальное ядро** агентской системы памяти. Оно одинаково для любого проекта и любого стека. Всё, что относится к конкретному стеку технологий, живёт в `STACK.md`. Карта конкретного репозитория — в `PROJECT_STRUCTURE.md`.

## Принцип: 4-документный цикл на задачу

Каждая нетривиальная задача документируется **четырьмя артефактами**, связанными одним общим `task-id`:

```text
spec  →  analysis  →  implementation log  →  final report
```

| Этап | Папка | Назначение |
|---|---|---|
| 1. Требования | `specs/` | Что нужно сделать. Цель, область, требования, ограничения, критерии приёмки. |
| 2. Исследование | `analysis/` | Что есть сейчас. Текущее поведение, релевантные файлы, риски, решения. |
| 3. Реализация | `implementation/` | Что сделано. Журнал изменённых файлов с причиной и побочными эффектами. |
| 4. Итог | `reports/` | Результат. Сводка, проверка, известные пробелы, следующие шаги. |

Параллельно работают два поддерживающих раздела:
- `decisions/` — архитектурные решения (ADR), выходящие за рамки одной задачи.
- `specs/backlog.md` — будущие задачи и идеи.

## Идентификаторы задач (task-id)

Используется два уровня:

- `0.x` — **служебные/мета-задачи**: касаются самой системы памяти, правил, инфраструктуры.
- `1.x`, `2.x`, … — **продуктовые задачи**: реальная функциональность проекта. Каждый крупный эпик/поток открывает новую старшую цифру.

`task-id` обязан быть **одним и тем же** во всех четырёх документах задачи.

Пример: задача `1.4` (установка модуля доставки) представлена файлами
`specs/1.4-measoft-install.md`, `analysis/1.4-measoft-install-analysis.md`,
`implementation/1.4-implementation-log.md`, `reports/1.4-final-report.md`.

## Имена файлов

| Папка | Шаблон | Пример |
|---|---|---|
| `specs/` | `<task-id>-<slug>.md` | `1.1-checkout-flow.md` |
| `analysis/` | `<task-id>-<slug>-analysis.md` | `1.1-current-checkout-analysis.md` |
| `implementation/` | `<task-id>-implementation-log.md` | `1.1-implementation-log.md` |
| `reports/` | `<task-id>-final-report.md` | `1.1-final-report.md` |
| `decisions/` | `ADR-NNN-<slug>.md` | `ADR-001-checkout-architecture.md` |

`<slug>` — короткое описательное имя в kebab-case (`checkout-flow`, `measoft-install`).

## Обязательные разделы каждого документа

Шаблоны с заготовками лежат в `templates/`. Краткая сводка:

- **spec**: Цель · Область · Требования · Ограничения · Критерии Приёмки.
- **analysis**: Текущее Поведение · Релевантные Файлы · Поток Данных · Риски · Решения · Архитектурные Решения · Результаты Анализа.
- **implementation log**: План · Изменения · Журнал Изменённых Файлов · Проверка · Открытые Пункты. Каждая запись об изменённом файле: **Файл · Причина · Краткое описание · Возможные побочные эффекты**.
- **final report**: Сводка · Изменённые Файлы · Проверка · Известные Пробелы · Следующие Шаги.
- **ADR**: Статус (Draft/Accepted/Superseded) · Контекст · Решение · Последствия.

## Заголовок документа задачи

Каждый документ задачи начинается с метаблока:

```text
**Задача:** <task-id>-<slug>
**Дата:** YYYY-MM-DD
**Статус:** в работе / завершён / код подготовлен
```

## Правила жизненного цикла

1. **Один task-id** на задачу во всех четырёх документах.
2. **Append-only по умолчанию**: не удалять существующую документацию, дополнять или редактировать разделы, сохраняя полезную историю.
3. **Если пользователь дал task-id или имя файла — использовать его.** Если нет — выбрать slug и сохранить нумерацию согласованной с существующими файлами.
4. **Не хранить секреты, токены, полные логи, данные клиентов, креды окружения** ни в одном документе.
5. **Документировать каждое изменение кода или документации** в `implementation/`.
6. **После завершения задачи** создавать или обновлять отчёт в `reports/` и запись в `INDEX.md`.

## Порядок загрузки контекста

Перед широкой работой читать в порядке:

1. `README.md` (этот раздел) — навигация.
2. `CORE.md` (этот файл) — как устроена память.
3. `PROJECT_STRUCTURE.md` — карта конкретного репозитория.
4. `STACK.md` — правила и команды конкретного стека.
5. `RULES.md` — универсальные правила.
6. `CHECKLISTS.md` — повторяемые чек-листы.
7. Документ задачи в `specs/` (если работаем по существующей задаче).

## Память: когда что обновлять

Перед завершением задачи обновить систему памяти, если найдено долговечное знание:

| Тип знания | Куда |
|---|---|
| Новая карта модулей, владение файлами, структура репозитория | `PROJECT_STRUCTURE.md` |
| Новые правила, ограничения, небезопасные пути, команды стека | `STACK.md` |
| Универсальное правило, не привязанное к стеку | `RULES.md` |
| Повторяемый шаг или процесс проверки | `CHECKLISTS.md` |
| Новая задача или изменение статуса | `INDEX.md` |
| Изменение самой архитектуры памяти | `CHANGELOG.md` |
| Архитектурное решение, выходящее за рамки задачи | `decisions/ADR-NNN-<slug>.md` |

## Что НЕ часть ядра

- **Стек-специфика** (пути OpenCart, artisan-команды Laravel, npm-скрипты) → `STACK.md`.
- **Карта конкретного репо** → `PROJECT_STRUCTURE.md`.
- **Скиллы и хуки** (graphify, кастомные команды) → `skills/` + `hooks.json`, опционально.
