# Универсальные Чек-листы

Последнее обновление: см. `CHANGELOG.md`

Эти чек-листы применяются в **любом** проекте. Стек-специфичные чек-листы (PHP/Blade, OpenCart OCMOD, React-компоненты и т.д.) — в `STACK.md`.

## Перед Любым Изменением Кода

1. Прочитать `.agent/README.md` и `.agent/CORE.md`.
2. Прочитать `.agent/PROJECT_STRUCTURE.md` и `.agent/STACK.md`.
3. Проверить `git status --short`.
4. Определить, затрагивает ли задача исходный код, конфиг, миграции, документацию или инфраструктуру.
5. Для новых нетривиальных задач определить `task-id` и создать/обновить файлы в `specs/`, `analysis/`, `implementation/`, `reports/`.
6. Просматривать только сфокусированные файлы, нужные для задачи.
7. Новые и обновляемые документы писать на русском, оставляя технические идентификаторы без перевода.

## Процесс Документирования Новой Задачи

1. Создать/обновить `specs/<task-id>-<slug>.md` (цель, область, требования, ограничения, критерии приёмки).
2. Создать/обновить `analysis/<task-id>-<slug>-analysis.md` (текущее поведение, релевантные файлы, поток данных, риски, решения, результаты анализа).
3. Обновлять `implementation/<task-id>-implementation-log.md` во время реализации значимых изменений.
4. Для каждого изменённого файла добавить запись: файл, причина, краткое описание, возможные побочные эффекты.
5. Перед завершением создать/обновить `reports/<task-id>-final-report.md` (сводка, изменённые файлы, проверка, известные пробелы, следующие шаги).
6. Поддерживать один task-id во всех четырёх документах.
7. Добавить/обновить запись в `INDEX.md`.
8. Дополнять существующую документацию вместо удаления.

## Перед Завершением Задачи

1. Обновлены ли 4 документа задачи (или явно указано, какие не нужны)?
2. Записан ли каждый изменённый файл в журнале реализации?
3. Запущены ли линтер/тесты стека (см. `STACK.md`)?
4. Для этого проекта не выполнялось ли локальное HTTP/браузерное тестирование сайта из Codex (проверка витрины только через git commit + серверный deploy)?
5. Обновлён ли `INDEX.md` (статус задачи)?
6. Обновлены ли долговечные знания (`PROJECT_STRUCTURE.md` / `STACK.md` / `RULES.md` / `CHECKLISTS.md` / ADR), если они обнаружены?
7. Нет ли случайно закоммиченных секретов, логов, данных клиентов?
8. Явно ли указано, что НЕ проверялось?

## Чек-лист Финального Ответа Пользователю

1. Что изменилось или что найдено.
2. Затронутые файлы со ссылками/путями, когда полезно.
3. Выполненная проверка.
4. Невыполненная проверка, если релевантно.
5. Пути `specs/` / `analysis/` / `implementation/` / `reports/` задачи, если они созданы или обновлены.
6. Только конкретные следующие шаги, естественно вытекающие из задачи.
