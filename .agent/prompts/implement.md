# Промт: Реализация Задачи

Этап: **analysis → implementation**. Внесение изменений в код и документирование.

## Вход

- `.agent/CORE.md` — конвенции документации.
- `.agent/PROJECT_STRUCTURE.md` — карта репозитория.
- `.agent/STACK.md` — правила и команды стека (важно: что предпочитать, что не трогать).
- `.agent/RULES.md` — универсальные правила (git, безопасность, секреты).
- `.agent/specs/<task-id>-<slug>.md` — требования.
- `.agent/analysis/<task-id>-<slug>-analysis.md` — исследование и план.

## Что Сделать

1. Проверить `git status --short` перед правками. Не откатывать чужие несвязанные изменения.
2. Реализовать требования согласно `STACK.md` (предпочитать исходные файлы, не трогать сгенерированные).
3. На каждый изменённый файл — запись в журнале: **Файл · Причина · Краткое описание · Возможные побочные эффекты**.
4. Не коммитить секреты/логи/данные клиентов.
5. Запустить линтер/тесты стека (см. `STACK.md`) для затронутой области.
6. Явно отметить, что НЕ было проверено.

## Куда Сохранить

`.agent/implementation/<task-id>-implementation-log.md` (скелет — `.agent/templates/implementation-template.md`).

Обязательные разделы: План · Изменения · Журнал Изменённых Файлов · Проверка · Открытые Пункты.

## После

- Если найдены новые долговечные знания — обновить `PROJECT_STRUCTURE.md` / `STACK.md` / `RULES.md` / `CHECKLISTS.md`.
- Если всплыло архитектурное решение — создать ADR.
- Кратко доложить: что изменено, какие файлы, что проверено, что требует ручных шагов.
- **Не коммитить**, если пользователь явно не попросил.
