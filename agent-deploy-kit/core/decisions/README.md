# Архитектурные Решения (ADR)

Здесь хранятся Architecture Decision Records — архитектурные решения, **выходящие за рамки одной задачи**. Если решение касается только одной задачи, фиксируйте его в её `analysis/` файле.

## Именование

- `ADR-NNN-<slug>.md`, где `NNN` — последовательный номер с ведущими нулями (`ADR-001`, `ADR-002`, …).
- Пример: `ADR-001-checkout-architecture.md`.
- Скелет нового файла — `../templates/adr-template.md`.

## Обязательные разделы

- **Статус**: `Draft` | `Accepted` | `Deprecated` | `Superseded by ADR-NNN`.
- **Контекст**: почему возникла необходимость в решении.
- **Решение**: что решили.
- **Последствия**: плюсы, минусы, риски, влияние на существующий код.

## Правила

- Один ADR — одно самостоятельное решение.
- Номер ADR никогда не переиспользуется. При отмене — меняйте статус, не удаляйте файл.
- При `Superseded` — создайте новый ADR с новым номером и ссылкой на старый.
- Регистрируйте новый ADR в `../INDEX.md` (таблица «Архитектурные Решения»).
- Основной текст — на русском; технические идентификаторы без перевода.

## Пример заголовка

```markdown
# ADR-001 <короткое название решения>

## Статус
Accepted

## Контекст
<почему>

## Решение
<что>

## Последствия
<эффекты>
```
