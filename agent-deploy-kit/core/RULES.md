# Универсальные Правила

Последнее обновление: см. `CHANGELOG.md`

Эти правила применяются в **любом** проекте. Стек-специфичные правила (команды, пути, соглашения фреймворка) — в `STACK.md`. Карта репозитория — в `PROJECT_STRUCTURE.md`.

## Загрузка Контекста

1. Перед широкой работой прочитать `README.md`, `CORE.md`, `PROJECT_STRUCTURE.md`, `STACK.md`.
2. **Не сканировать весь репозиторий рекурсивно**, если задача этого не требует.
3. Сначала использовать точечный поиск в вероятных исходных путях из `PROJECT_STRUCTURE.md`.
4. По умолчанию избегать сгенерированных, больших и чувствительных директорий — список см. в `STACK.md` (раздел «Пути, по умолчанию не сканировать») и `PROJECT_STRUCTURE.md`.

## Язык Документации

1. Все новые и обновляемые документы в `.agent` писать **на русском языке**.
2. Спецификации, анализ, журналы реализации, итоговые отчёты, чек-листы и проектные правила — на русском.
3. **Технические идентификаторы не переводить**: пути, имена файлов, классы, методы, маршруты, коды расширений, переменные, команды, SQL, XML, PHP/Blade/Twig/JS-фрагменты и значения конфигурации оставлять в оригинальном виде.
4. Внешние названия (`CDEK`, `Measoft`, `OpenCart`, `PayPal`, `Laravel`, `Stripe`) оставлять как есть.
5. Основной текст, заголовки разделов и пояснения новых документов должны быть русскими.

## Структура Документации Задач

Подробно — в `CORE.md`. Кратко:

1. Использовать **один общий task-id** во всех документах задачи.
2. Требования → `specs/<task-id>-<slug>.md`.
3. Исследование → `analysis/<task-id>-<slug>-analysis.md`.
4. Ход реализации → `implementation/<task-id>-implementation-log.md`.
5. Итог → `reports/<task-id>-final-report.md`.
6. Каждая запись журнала реализации для изменённого файла включает: файл, причина, краткое описание, возможные побочные эффекты.
7. Не удалять существующую документацию — дополнять или редактировать разделы.
8. Не хранить секреты, токены, полные логи, данные клиентов, креды окружения.

## Git И Существующие Изменения

1. Перед правками кода проверять `git status --short`.
2. **Не откатывать существующие изменения пользователя**, если это явно не запрошено.
3. Считать несвязанные неотслеживаемые или изменённые файлы принадлежащими пользователю.
4. Предпочитать неинтерактивные команды Git.
5. Новые сущности системы памяти (файлы `.agent/`) — коммитить отдельно от кодовых изменений, чтобы история была читаемой.
6. При переносе/переименовании использовать `git mv`, чтобы сохранить историю.

## Безопасность

1. Считать конфиги, токены, логи и админские инструменты чувствительными.
2. **Не раскрывать реальные секреты** в новой документации или примерах. Использовать плейсхолдеры (`<API_KEY>`, `***`).
3. Не добавлять новые публичные скрипты, изменяющие состояние БД, без аутентификации.
4. Файлы `.env`, `*.token`, `config*.php` с кредами — никогда не копировать в документы дословно.
5. Если найден раскрытый секрет — отметить это в отчёте и рекомендовать ротацию.

## Проверка

1. Запускать линтер/тайп-чекер стека (см. `STACK.md`) для изменённых файлов.
2. Запускать тесты стека, если они есть и релевантны изменению.
3. Для изменений внешнего поведения — по возможности проверять полный поток вручную или чётко указывать, что не проверялось.
4. В отчёте явно указывать **выполненную** и **невыполненную** проверку.

## Память Системы Перед Завершением

Перед завершением задачи обновить систему памяти, если найдено долговечное знание (подробная таблица — в `CORE.md`):

- Новая карта модулей/владение файлами → `PROJECT_STRUCTURE.md`.
- Новые правила/ограничение/команда стека → `STACK.md`.
- Универсальное правило → `RULES.md`.
- Повторяемый шаг → `CHECKLISTS.md`.
- Архитектурное решение → `decisions/ADR-NNN-<slug>.md`.
- Статус задачи → `INDEX.md`.

## Финальный Ответ Пользователю

1. Указать, что изменилось или что найдено.
2. Упомянуть затронутые файлы со ссылками/путями, когда полезно.
3. Указать выполненную проверку.
4. Указать невыполненную проверку, если релевантно.
5. Указать пути `specs/`, `analysis/`, `implementation/`, `reports/` задачи, если они созданы или обновлены.
6. Предлагать только конкретные следующие шаги, естественно вытекающие из задачи.
